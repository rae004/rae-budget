# Plan: Replace SQL init with Alembic migrations

## Goal & Context

Today the schema is bootstrapped by `db/init/01_init.sql` running once on first PostgreSQL volume init. That works for fresh installs but has no answer for schema evolution ‚Äî every column add requires either a manual `ALTER TABLE` or destroying the volume. Move to **Alembic** so schema changes ride along in version control and apply automatically on container start.

User decisions:
- **Existing DBs**: stamp as baseline (preserve data ‚Äî your dev DB stays intact)
- **Seeds**: drop the 10 default categories; fresh installs start with no data

---

## Approach

- Standard **sync Alembic** (matches the codebase's sync SQLAlchemy 2.x style)
- Alembic lives at `backend/alembic.ini` + `backend/alembic/versions/`
- `env.py` imports `Base` from `app.extensions` and all models from `app.models` so autogenerate works
- API container runs a small `migrate.py` script before `flask run` ‚Äî auto-detects pre-existing schema without an `alembic_version` table and stamps it before applying migrations
- `db/init/` is removed entirely (file + directory + docker-compose mount)

## Implementation steps

### Step 1 ‚Äî Add Alembic dependency

```bash
docker compose exec api uv add alembic
```

(Or edit `pyproject.toml` directly and rebuild.)

### Step 2 ‚Äî Initialize Alembic structure

```bash
docker compose exec api alembic init alembic
```

This creates `backend/alembic.ini` and `backend/alembic/{env.py, script.py.mako, versions/}`.

### Step 3 ‚Äî Customize `env.py`

Replace the generated `env.py` so it:
1. Reads `DATABASE_URL` from environment (not hard-coded in `alembic.ini`)
2. Imports `Base` from `app.extensions`
3. Imports all models from `app.models` (forces SQLAlchemy to register them on `Base.metadata` for autogenerate)
4. Uses `target_metadata = Base.metadata`

Key snippets:

```python
import os
from logging.config import fileConfig
from sqlalchemy import engine_from_config, pool
from alembic import context

# Make app importable
import sys
sys.path.insert(0, os.path.abspath(os.path.join(os.path.dirname(__file__), "..")))

from app.extensions import Base
from app import models  # noqa: F401 ‚Äî side-effect: registers models on Base.metadata

config = context.config

# Read DB URL from env, not alembic.ini
db_url = os.environ.get("DATABASE_URL")
if db_url:
    config.set_main_option("sqlalchemy.url", db_url)

if config.config_file_name is not None:
    fileConfig(config.config_file_name)

target_metadata = Base.metadata

# ... rest of standard env.py (run_migrations_offline / run_migrations_online)
```

### Step 4 ‚Äî Generate the baseline migration

```bash
docker compose exec api alembic revision --autogenerate -m "baseline schema"
```

Review the generated migration ‚Äî it should create all 5 tables, indexes, FK constraints, and `updated_at` triggers. **One gotcha**: Alembic's autogenerate doesn't emit raw SQL like the `update_updated_at_column()` PL/pgSQL function or its triggers. Either:
- (a) Add them manually to the baseline migration's `upgrade()` as `op.execute("CREATE OR REPLACE FUNCTION ...")` blocks, or
- (b) Drop the triggers entirely and rely on SQLAlchemy's `default=lambda: datetime.now(UTC)` + `onupdate=...` at the model layer

**Recommended: (b)** ‚Äî the model already sets `created_at` defaults; just add `onupdate=lambda: datetime.now(UTC)` to `updated_at` columns where it's missing. Removes the SQL-side trigger machinery entirely. Cleaner.

### Step 5 ‚Äî Add `backend/scripts/migrate.py`

```python
"""Run Alembic migrations, stamping pre-existing schemas as baseline first."""
import os
import sys
from alembic import command
from alembic.config import Config
from sqlalchemy import create_engine, inspect


def main() -> None:
    cfg = Config("alembic.ini")
    db_url = os.environ["DATABASE_URL"]
    cfg.set_main_option("sqlalchemy.url", db_url)

    engine = create_engine(db_url)
    inspector = inspect(engine)
    table_names = set(inspector.get_table_names())

    has_app_tables = bool(table_names & {"categories", "pay_periods"})
    has_alembic_version = "alembic_version" in table_names

    if has_app_tables and not has_alembic_version:
        print("[migrate] Pre-existing schema detected; stamping baseline...", flush=True)
        command.stamp(cfg, "head")

    print("[migrate] alembic upgrade head", flush=True)
    command.upgrade(cfg, "head")


if __name__ == "__main__":
    sys.exit(main() or 0)
```

### Step 6 ‚Äî Re-add `backend/entrypoint.sh`

```bash
#!/bin/bash
set -e

cd /app
python scripts/migrate.py
exec "$@"
```

### Step 7 ‚Äî Update `Dockerfile`

Add at the bottom (before `CMD`):

```dockerfile
COPY entrypoint.sh /entrypoint.sh
RUN chmod +x /entrypoint.sh

ENTRYPOINT ["/entrypoint.sh"]
```

`CMD ["flask", "run", "--host=0.0.0.0", "--port=5000"]` stays as the default args passed to `entrypoint.sh`'s `exec "$@"`.

### Step 8 ‚Äî Update `docker-compose.yml`

Remove the init-script mount on `postgres`:

```diff
     volumes:
       - postgres_data:/var/lib/postgresql/data
-      - ./db/init:/docker-entrypoint-initdb.d:ro
```

### Step 9 ‚Äî Delete `db/init/` and rebuild

```bash
rm -rf db/init/
docker compose build api
docker compose up -d
```

## How existing DBs are handled (your dev DB)

On first restart with the new image:
1. `entrypoint.sh` runs `migrate.py`
2. `migrate.py` inspects: tables exist (`categories`, `pay_periods`, ‚Ä¶) but no `alembic_version` table
3. Detects pre-existing state ‚Üí `alembic stamp head` writes the baseline revision into a new `alembic_version` table
4. `alembic upgrade head` runs ‚Äî it's already at head, no-op
5. Flask starts normally
6. Your data is untouched ‚úÖ

On a fresh install (new volume):
1. Postgres starts empty
2. `migrate.py` finds no tables ‚Üí skips stamping
3. `alembic upgrade head` creates the entire schema from the baseline migration
4. Flask starts; user is greeted with empty tables (no seeds, per request)

## Workflow going forward

When you change a model (add/remove/modify a column):

```bash
docker compose exec api alembic revision --autogenerate -m "add X to Y"
# review the generated file in backend/alembic/versions/
git add backend/alembic/versions/<file>.py
docker compose restart api  # entrypoint reapplies ‚Äî your local DB is now upgraded
```

Other devs `git pull` and `docker compose up -d` ‚Äî their entrypoint runs the new migration on container start. No coordination needed.

Useful commands to document in the README's Development section:

```bash
docker compose exec api alembic current             # what version am I on?
docker compose exec api alembic history             # show all revisions
docker compose exec api alembic downgrade -1        # roll back one revision
docker compose exec api alembic upgrade head        # apply pending migrations
```

## Verification

1. **Existing DB preservation**: with your current volume, `docker compose down && docker compose up -d`, then check `\dt` in psql shows all tables intact AND `alembic_version` exists with one row matching the baseline revision id.
2. **Fresh install**: `docker compose down -v && docker compose up -d`. Check that the schema is created with all 5 tables, all indexes, all FK constraints. Categories table is empty (no seeds).
3. **New migration smoke test**: add a dummy column to a model, run `alembic revision --autogenerate -m "test col"`, verify the generated migration upgrades cleanly. Then `alembic downgrade -1` to verify the downgrade works. Revert the dummy change.
4. **Tests still pass**: `docker compose exec api uv run pytest` ‚Äî should be unaffected since tests use in-memory SQLite via SQLAlchemy's `Base.metadata.create_all` (per `conftest.py`), not Alembic.
5. **CI**: backend tests don't touch Postgres or Alembic, so no CI changes needed.
